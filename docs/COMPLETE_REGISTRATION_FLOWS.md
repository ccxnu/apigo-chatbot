# Complete Registration Flows - All Scenarios

## Overview

The system now supports THREE types of users:
1. **Institute Students** (found in AcademicOK apidatospersona)
2. **Institute Professors** (found in AcademicOK apidatosdocente)
3. **External Users** (NOT in AcademicOK - visitors, partners, etc.)

All require **email OTP verification** to complete registration.

## Edge Case: Registered User Sends "reenviar" ‚ùå‚Üí‚úÖ

### Scenario
```
User: "reenviar"  [Already registered user]
```

### What Happens

**1. RegistrationHandler checks**
```go
func (h *RegistrationHandler) Match(ctx, msg) bool {
    result := h.userUseCase.GetUserByWhatsApp(ctx, msg.From)
    if result.Success && result.Data != nil {
        return false  // ‚ùå Handler doesn't match
    }
    return true
}
```

**2. Handler doesn't match ‚Üí Falls through to next handler**
- Message goes to CommandHandler or RAGHandler
- User gets normal response (RAG answer or "command not found")

**3. No error or confusion** ‚úÖ
- System gracefully handles registered users
- They can use chatbot normally

## Edge Case: OTP Expiration During External Registration

### Scenario
```
External user provides email ‚Üí receives OTP ‚Üí waits 15 minutes ‚Üí sends OTP
```

### What Happens

**1. User sends expired OTP**
```
User: "789012"  [Code is > 10 minutes old]
```

**2. System verifies**
```sql

-- Returns: success=FALSE, code='ERR_OTP_EXPIRED'
```

**3. Bot responds**
```
Bot: ‚è∞ El c√≥digo ha expirado. Escribe 'reenviar' para generar un nuevo c√≥digo.
```

**4. User requests new OTP**
```

```

**5. System regenerates OTP**
- Generates new 6-digit code
- Updates `pnd_otp_code` and `pnd_otp_expires_at`
- Sends new email
- Old code is invalid

## Security Benefits

### For Institute Users
- ‚úÖ Cedula validated against official AcademicOK database
- ‚úÖ Email verified (must have access to institutional email)
- ‚úÖ Cannot register with someone else's cedula + phone combination

### For External Users
- ‚úÖ Email verified (must have access to provided email)
- ‚úÖ Cannot impersonate institute members (cedula not in database)
- ‚úÖ Clearly marked as "external" for auditing purposes
- ‚úÖ Same OTP security as institute users

## Testing Checklist

### Institute Student
- [ ] Valid cedula ‚Üí OTP sent ‚Üí Correct OTP ‚Üí Registered ‚úÖ
- [ ] Valid cedula ‚Üí OTP sent ‚Üí Wrong OTP ‚Üí Error + retry
- [ ] Valid cedula ‚Üí OTP sent ‚Üí Expired OTP ‚Üí Resend option
- [ ] Valid cedula ‚Üí OTP sent ‚Üí "reenviar" ‚Üí New OTP

### Institute Professor
- [ ] Valid professor cedula ‚Üí OTP sent ‚Üí Registered as professor ‚úÖ
- [ ] Same OTP flow as students

### External User
- [ ] Invalid cedula ‚Üí Request email ‚Üí Provide name/email ‚Üí OTP sent ‚Üí Registered ‚úÖ
- [ ] Invalid cedula ‚Üí Request email ‚Üí Invalid format ‚Üí Re-ask
- [ ] Invalid cedula ‚Üí Request email ‚Üí Valid format ‚Üí OTP ‚Üí Verify ‚Üí Success

### Edge Cases
- [ ] Registered user sends "reenviar" ‚Üí Falls through to other handlers ‚úÖ
- [ ] External user provides only email (no name) ‚Üí Re-ask
- [ ] External user provides malformed email ‚Üí Re-ask
- [ ] Any user exceeds 5 OTP attempts ‚Üí Must request new OTP
